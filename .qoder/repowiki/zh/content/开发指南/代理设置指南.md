## 简介

本指南专门针对 AutoJS6 Visual Development Toolkit 项目中的 GitHub 代理设置问题，提供了完整的解决方案。当开发者的网络环境无法直接访问 GitHub 时，以下任务可能会失败：

- `git clone` 克隆仓库
- `git fetch / pull / push` 拉取、拉取和推送操作
- 在浏览器中打开 GitHub
- 将 `.github/workflows/*` 推送到 GitHub 后验证 Actions

## 适用场景

### 何时需要代理设置

如果您的网络环境无法直接访问 GitHub，以下情况都会受到影响：

- **Git 操作失败**：克隆、推送、拉取等基本 Git 操作
- **浏览器访问受限**：无法打开 GitHub 网站
- **GitHub Actions 无法显示**：即使本地有工作流文件，GitHub 页面也可能看不到

### 重要提醒

> **如果 workflow 文件还没有真正 push 到 GitHub 默认分支，GitHub Actions 页面可能完全看不到对应的工作流。**
> 所以"先解决代理，确保能 push 到 GitHub"，是使用 GitHub Actions 的前置条件。

## 代理配置方案

### 推荐默认方案

对于大多数开发者，最简单的做法是：

1. **切换 GitHub 远端到 HTTPS**
2. **配置 Git 代理**

这种方法通常比在代理环境下保持 SSH 远端更容易调试。

### 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| HTTPS + Git 代理 | 配置最简单、最容易排查、对大多数代理软件友好 | 需要额外的 Git 代理配置 | 大多数开发者 |
| SSH 代理 | 保持 SSH 连接特性 | 需要额外的 SSH 代理工具、配置复杂 | 有特殊需求的高级用户 |

## HTTPS 远端配置

### 临时 PowerShell 会话代理

如果您的代理提供 SOCKS5 端口（如 `127.0.0.1:12334`），在当前 PowerShell 会话中设置：

```powershell
$env:HTTP_PROXY="socks5h://127.0.0.1:12334"
$env:HTTPS_PROXY="socks5h://127.0.0.1:12334"
$env:ALL_PROXY="socks5h://127.0.0.1:12334"
```

如果您的代理提供 HTTP 端口（如 `127.0.0.1:7890`）：

```powershell
$env:HTTP_PROXY="http://127.0.0.1:7890"
$env:HTTPS_PROXY="http://127.0.0.1:7890"
```

### 直接测试网页连通性

建议使用 `curl.exe`，避免 PowerShell 别名混淆：

```powershell
curl.exe --proxy socks5h://127.0.0.1:12334 https://github.com
```

如果使用 HTTP 代理：

```powershell
curl.exe --proxy http://127.0.0.1:7890 https://github.com
```

### 切换到 HTTPS 远端

如果当前使用 SSH 远端，可以改为 HTTPS：

```powershell
git remote set-url origin https://github.com/terwer/autojs6-dev-tools.git
```

然后验证：

```powershell
git remote -v
```

确保 `origin` 已经变成 `https://github.com/...`

### Git 代理配置

#### 全局配置

```powershell
git config --global http.proxy socks5h://127.0.0.1:12334
git config --global https.proxy socks5h://127.0.0.1:12334
```

如果使用 HTTP 代理端口：

```powershell
git config --global http.proxy http://127.0.0.1:7890
git config --global https.proxy http://127.0.0.1:7890
```

#### 仅对当前仓库配置

```powershell
git config --local http.proxy socks5h://127.0.0.1:12334
git config --local https.proxy socks5h://127.0.0.1:12334
```

### 验证 GitHub 连通性

先进行基本测试：

```powershell
git ls-remote origin
```

再测试推送前的基本链路：

```powershell
git fetch origin
```

如果这两步都成功，再进行正常的 `push` 操作。

## SSH 远端配置

### 保留 SSH 远端的原因

如果您坚持继续使用 SSH 远端：

```text
git@github.com:terwer/autojs6-dev-tools.git
```

需要注意：**仅设置 `HTTP_PROXY` / `HTTPS_PROXY` 不一定能使 SSH Push 生效。**

这是因为 SSH 走的是另一套网络链路。

### 需要额外配置 SSH 代理

编辑：

```text
C:\Users\<你的用户名>\.ssh\config
```

示例（使用 `ncat` 走 SOCKS5）：

```sshconfig
Host github.com
  HostName github.com
  User git
  ProxyCommand ncat --proxy 127.0.0.1:12334 --proxy-type socks5 %h %p
```

如果还有其他 GitHub 仓库，也可以继续复用这个配置。

### 验证 SSH 是否可用

```powershell
ssh -T git@github.com
```

如果能握手成功，继续：

```powershell
git ls-remote origin
git fetch origin
git push origin <branch>
```

### 无 `ncat` 时的替代方案

如果本机没有 `ncat`，最简单的替代方案不是继续折腾 SSH，而是：

> **直接切回 HTTPS 远端。**

对这个项目来说，这是更节省时间的方案。

## 验证流程

### 推荐的验证顺序

1. `curl.exe --proxy ... https://github.com`
2. `git remote -v`
3. `git ls-remote origin`
4. `git fetch origin`
5. `git push origin <branch>`
6. 到 GitHub 网页确认分支更新
7. 再去看 Actions 页面是否出现 workflow

### 推送完成后要做什么

如果这次推送的是：

- `.github/workflows/manual-release-test.yml`
- `.github/workflows/release-please.yml`

那么推送完成后要去 GitHub Actions 页面查看。

否则即使本地有 workflow 文件，GitHub 页面仍然可能看不到。

## 常见问题与解决方案

### 已设置 PowerShell 代理，为什么 `git push` 仍不通？

因为 `origin` 很可能仍然是：

```text
git@github.com:...
```

也就是 SSH 远端。这时 HTTP 代理变量未必能自动接管 SSH。

### `curl https://x.com` 不通，说明 GitHub 一定也不通？

不一定，但通常说明当前代理链本身就没打通。建议先把 `curl.exe --proxy ... https://github.com` 跑通，再测 Git。

### GitHub Actions 页面没显示 workflow，是 GitHub 缓存？

很多时候不是缓存，而是：

- workflow 文件还没 push 到 GitHub
- workflow 文件不在默认分支
- push 根本没成功

所以第一步不是刷新页面，而是先验证：

```powershell
git push origin <branch>
```

确认是否真的成功了。

## GitHub Actions 集成

### 手动发布测试工作流

项目包含两个主要的 GitHub Actions 工作流：

#### manual-release-test 工作流

这个工作流用于手动发布测试，支持多种输入参数：

```yaml
name: manual-release-test
on:
  workflow_dispatch:
    inputs:
      source_ref: 代码来源
      version: 版本号
      publish_to_release: 是否发布到 GitHub Release
      release_tag: Release 标签
      release_name: Release 标题
      prerelease: 是否预发布
```

#### release-please 工作流

这个工作流用于自动发布管理：

```yaml
name: release-please
on:
  push:
    branches:
      - main
```

### 发布流程

#### 本地发布验证顺序

使用以下顺序在本地验证发布候选版本：

1. `dotnet restore autojs6-dev-tools.slnx`
2. `dotnet build autojs6-dev-tools.slnx -c Release`
3. `dotnet test autojs6-dev-tools.slnx -c Release`
4. 构建 `win-x64` 和 `win-arm64` 便携式 ZIP 包
5. 运行 `scripts/release/Test-PortablePackageSmoke.ps1` 针对 `win-x64` 便携式 EXE
6. 构建 `win-x64` 和 `win-arm64` EXE 安装程序
7. 构建 `win-x64` 和 `win-arm64` MSIX 包
8. 确认 `release-assets/` 中的文件与预期版本、发布者和 SHA256 列表匹配

### 发布脚本

#### 设置应用发布元数据

```powershell
./scripts/release/Set-AppReleaseMetadata.ps1
    -Version "${{ steps.meta.outputs.version }}"
    -ProductName "${env:PRODUCT_NAME}"
    -PackageIdentityName "${env:PACKAGE_IDENTITY_NAME}"
    -Publisher "${env:PUBLISHER}"
    -PublisherDisplayName "${env:PUBLISHER_DISPLAY_NAME}"
```

#### 构建便携式包

```powershell
./scripts/release/Build-PortablePackage.ps1
    -ProjectPath "App/App.csproj"
    -RuntimeIdentifier "win-x64"
    -Version "${{ steps.meta.outputs.version }}"
    -OutputRoot "artifacts"
```

## 故障排除指南

### 取消代理设置

#### 清空当前 PowerShell 会话环境变量

```powershell
Remove-Item Env:HTTP_PROXY -ErrorAction SilentlyContinue
Remove-Item Env:HTTPS_PROXY -ErrorAction SilentlyContinue
Remove-Item Env:ALL_PROXY -ErrorAction SilentlyContinue
```

#### 删除 Git 全局代理

```powershell
git config --global --unset http.proxy
git config --global --unset https.proxy
```

#### 删除 Git 当前仓库代理

```powershell
git config --local --unset http.proxy
git config --local --unset https.proxy
```

### 代理配置完成后的最终验证

1. **网络连通性测试**：确保 `curl.exe --proxy ... https://github.com` 能正常访问
2. **Git 远端检查**：确认 `git remote -v` 显示正确的远程地址
3. **Git 基础功能**：验证 `git ls-remote origin` 和 `git fetch origin` 的可用性
4. **推送测试**：执行 `git push origin <branch>` 并确认 GitHub 页面更新
5. **Actions 验证**：最后检查 GitHub Actions 页面是否显示相应的工作流

## 最佳实践

### 快速解决策略

如果当前只是为了尽快把项目推到 GitHub 并让 Actions 可见，推荐优先采用：

1. **把 `origin` 改成 HTTPS**
2. **给 Git 配置代理**
3. **先确认 `git push origin <branch>` 成功**
4. **再回头验证 GitHub Actions**

这样通常比继续折腾 SSH 代理快得多。

### 代理设置经验建议

#### HTTP vs SOCKS5 选择

> **如果本机代理同时提供 HTTP 和 SOCKS5 端口，Git/curl 在 Windows 上优先尝试 HTTP 端口，通常更稳定。**

#### 代理配置优先级

1. **先测试网络连通性**：使用 `curl.exe` 测试代理是否正常工作
2. **确认 Git 远端类型**：区分 HTTPS 和 SSH 远端的不同处理方式
3. **验证 Git 基础功能**：确保 `ls-remote` 和 `fetch` 能正常工作
4. **最后进行推送测试**：确认 GitHub Actions 能正确识别工作流

通过遵循这些步骤和最佳实践，您应该能够成功解决 GitHub 访问问题，并确保项目的 GitHub Actions 能够正常工作。
