## 简介
本文件面向 AutoJS6 开发工具的持续集成与持续部署（CI/CD）流程，系统梳理 GitHub Actions 工作流配置与多平台打包策略。重点覆盖以下方面：
- 手动发布测试流程与自动化测试并行推进
- 多平台构建（x64、ARM64）与多种安装包格式（ZIP、MSIX、Inno Setup）
- 部署配置与证书管理，确保安全可靠的自动化发布
- CI/CD 管道各阶段职责划分：代码检出、环境准备、构建、测试、打包与发布
- **新增** CI 无交互安全指南，确保自动化流程在无头环境中稳定运行

## 核心组件
- GitHub Actions 工作流
  - release-please：自动创建/更新发布 PR，并在确认后完成正式发布
  - manual-release-test：按需触发的手动测试打包流程，用于验证与修复现有发布
- 打包与安装器
  - Inno Setup 安装器：生成 EXE 安装包
  - MSIX 包：Windows 应用商店兼容包
  - ZIP 便携包：直接解压运行
- 测试体系
  - 单元测试：App.Tests、Core.Tests、App.Tests/Core.Tests
  - 自动化测试：配合打包流程进行端到端验证
- **新增** CI 安全策略
  - 非交互式证书管理：自动跳过可能阻塞 CI 的信任存储导入
  - 脚本交互性审查：确保所有发布脚本在无头环境中可安全运行

## 详细组件分析

### 手动发布测试流程（manual-release-test）
该流程用于：
- 对主分支或已发布标签进行快速验证
- 仅在需要时生成完整安装包，避免资源浪费
- 支持"仅验证"与"修复现有发布"的两种模式

关键行为
- 选择源码分支/标签
- 选择版本号（建议使用非正式语义化版本）
- 控制是否上传到 GitHub Release
- 产出 ZIP、EXE、MSIX 三类安装包

```mermaid
flowchart TD
Start(["开始"]) --> ChooseSrc["选择源码来源<br/>main 或已有发布标签"]
ChooseSrc --> SetVer["设置版本号<br/>如 0.0.1 / 0.1.0"]
SetVer --> Mode{"模式选择"}
Mode --> |仅验证| NoUpload["关闭上传到 Release"]
Mode --> |修复发布| Upload["允许上传到 Release"]
NoUpload --> BuildAll["构建 ZIP/EXE/MSIX"]
Upload --> BuildAll
BuildAll --> TestRun["运行自动化测试"]
TestRun --> Verify{"验证通过？"}
Verify --> |否| Fix["修复问题并重试"]
Verify --> |是| Done(["结束"])
```

## CI 无交互安全指南

### 非交互式 CI 安全审查
发布路径现已专门审查"是否会阻塞等待人类操作"的行为，确保 CI 环境的稳定性。

#### 已确认高风险交互点
- **证书生成与导入脚本**
  - 风险来源：将生成的证书导入 TrustedPeople / Root
  - 为什么可能阻塞：Root 信任导入可能触发 Windows 信任确认，在无头 CI 中导致停滞
  - 当前行为：
    - CI/非交互会话自动跳过信任存储导入
    - 本地信任导入仅在显式传递参数时启用
    - 仅当显式传递以下参数时才启用：
      - `-ImportToTrustedPeople`
      - `-ImportToRoot`

#### 当前发布策略
- **MSIX 打包在 CI 中临时禁用**
  - manual-release-test 和 release-please 中均禁用
  - 当前支持的发布输出：
    - ZIP 便携包
    - EXE 安装器
  - MSIX 作为后续跟进项目，直到证书信任、签名验证、安装流程和最终用户体验得到端到端验证

#### 已审查的发布脚本（无需人工确认）
- **便携包构建脚本**
  - 使用 dotnet publish
  - 脚本本身无交互提示路径
- **Inno Setup 安装器构建脚本**
  - 使用 ISCC.exe
  - 无脚本级确认提示
  - 若缺少 Inno Setup，应快速失败并给出直接错误
- **MSIX 包构建脚本**
  - 目前保留在仓库中供未来使用
  - 不属于活动的 CI 发布路径
- **发布元数据设置脚本**
  - 仅进行文件编辑，无交互路径
- **便携包烟雾测试脚本**
  - 启动应用，等待有限秒数后强制停止
  - 这是时间限制的，不是无限确认等待

#### 不应在 CI 中使用的本地安装脚本
- **应用包生成的安装脚本**
  - App/AppPackages/**/Add-AppDevPackage.ps1
  - App/AppPackages/**/Install.ps1
这些生成的安装脚本适用于本地侧载/本地包安装流程，可能涉及证书信任或安装时交互。它们**不是** CI 发布打包路径的一部分，不应从 GitHub Actions 调用。

#### 工作流看起来"卡住"时的快速诊断
如果 Windows 发布作业出现冻结：

1. 检查**最后打印的日志行**
2. 识别确切的脚本/步骤名称
3. 假设最后一个可见操作是第一个可疑对象
4. 如果明显在等待信任/安装/UI 操作，取消运行
5. 在重新运行前修复脚本使其成为快速失败或非交互式

#### 实用规则
> 在 CI 中，允许包生成；机器信任更改不应是安全的默认值。

## 故障排查指南
- 手动测试失败
  - 优先修复代码、打包脚本与工作流配置问题
- 生产发布缺失文件
  - 修复现有发布而非新建版本，重新针对同一标签运行 manual-release-test 并回传缺失资产
- 生产包可用性问题
  - 修复主分支问题并发布补丁版本，避免重写已发布标签
- 本地 dotnet 构建失败
  - 检查平台目标、裁剪与 ReadyToRun 设置
- 本地 MSIX 签名错误
  - 核对证书主题与发布者、Signtool 可用性与本地证书导入状态
- 本地 EXE 安装器失败
  - 检查 Inno Setup 与输出路径权限
- **新增** CI 工作流挂起问题
  - 检查最后日志行，识别具体脚本/步骤
  - 如果是证书导入或安装确认，立即取消并修复为非交互式
  - 验证脚本的交互性审查结果
- **新增** MSIX 相关问题
  - 确认 MSIX 打包在 CI 中已被临时禁用
  - 等待证书信任验证完成后重新启用

## 结论
通过 manual-release-test 与 release-please 的双轨流程，AutoJS6 开发工具实现了"先验证、后发布"的稳健 CI/CD 实践。结合多平台与多格式打包策略，配合严格的证书与签名管理，确保了从开发到发布的全流程可控、可追溯与可复现。**新增的 CI 无交互安全指南进一步强化了自动化流程的稳定性，通过脚本交互性分析和故障排查机制，确保发布流程在各种环境中都能可靠运行。**
